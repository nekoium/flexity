# Product

## Register

product

## Users

Knowledge workers — individuals organizing notes, projects, and ideas into composable structures. Their context is everyday information work: capturing, rearranging, and reinterpreting the same underlying material as outlines, trackers, or SOPs depending on the task at hand. The main job is to turn scattered information into a single recursively composable model that can be re-viewed without re-entering data.

## Product Purpose

Flexity is a JSON-native, markdown-first information model for knowledge management, inspired by the Kanban system. It provides a structured way to organize information into recursively composable objects that can be viewed as knowledge outlines, idea or project trackers, SOPs, or other interpretations without changing the underlying data.

The docs site showcases that workflow: how one composable model becomes many views. Success looks like a visitor understanding the model by seeing it work — not by reading a spec alone.

## Brand Personality

Calm, clear, capable. Quiet confidence. The interface gets out of the way of the data; it never fuzzy, glassy, or decorative for its own sake. Welcoming enough that knowledge workers of any technical depth feel invited in; precise enough that the underlying JSON/markdown model is never hidden.

## Anti-references

- **Glassmorphism and decorative blur.** Frosted cards, gradient overlays, glassy panels. The model is the product; fuzzy layers between user and data are wrong.
- **Cluttered enterprise dashboards.** Dense toolbars, feature overload, SaaS-by-the-pound. Flexity is one model with many views, not every feature crammed into one screen.
- **Overly playful / animated.** Bouncy motion, mascots, gamified consumer tone. Undercuts the capable, tool-like trust the product needs.

## Design Principles

- **Show the model working.** The surface itself demonstrates composable objects becoming outlines, trackers, and SOPs. Don't describe the model — let visitors watch it re-view.
- **Clear over clever.** Never obscure the underlying JSON/markdown structure. What you see is what the data is.
- **Calm confidence.** Capable, not loud. A tool you trust, not one that demands attention.
- **Welcome first.** Knowledge workers of varying technical depth should feel invited in. Simple and clear beats sharp and intimidating.
- **Honest structure.** No decorative fuzzy/glassy layers between user and model. The interface is the model, made legible.

## Accessibility & Inclusion

Standard best practices where they apply: 4.5:1 body text contrast, keyboard navigation for interactive elements, and `prefers-reduced-motion` respected.

The dashboard is inherently a visual management system (Kanban-style columns, cards, spatial arrangement) and is not designed — and realistically cannot serve well — for sight-impaired users. Accessibility effort focuses on low-vision users, colorblind-safe palette choices, and motion reduction rather than full screen-reader support for the visual surface.
